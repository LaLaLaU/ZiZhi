# Answer Generation Spec

## 1. 目标

基于结构化问题状态与历史证据，输出“像幕僚报告”的可执行回答，而不是泛泛而谈的鸡汤式建议。

## 2. 输入

回答生成节点至少接收：

- 结构化问题理解结果
- 检索证据集合
- 历史镜像映射结果
- 策略草稿
- 反思节点反馈
- 情绪标签

当前相关代码：

- `zizhi/agents.py`
- `zizhi/workflow.py`
- `zizhi/schemas.py`

## 3. 输出结构

正式输出建议固定为以下模块：

- `局势概述`
- `关键矛盾`
- `历史镜像`
- `策略建议`
- `风险提醒`
- `24小时动作`
- `7天动作`
- `Mermaid 博弈图`
- `情绪安抚（可选）`

## 4. 历史证据使用规则

- 每条关键判断尽量对应 1 条以上史例支撑
- 历史证据必须说明“相似点是什么”，不能只贴引文
- 若证据更像反例，应标记为 `contrast` 或 `caution`
- 不得把史例写成“历史已经证明你必须这样做”

## 5. 原文展示规则

- 主回答正文以现代白话表达为主
- 古文原文主要用于证据卡片或引用区展示
- 原文展示通过 `section_key` 回查，不直接依赖主生成文本携带全部原文
- 若原文过长，默认摘录关键句，不全量铺开

## 6. 置信度与边界

- 对用户单方叙述，默认保留假设边界
- 对证据不足的问题，明确写出“需进一步验证”的部分
- 当多个策略都可行时，输出 `A/B/C` 路径而不是伪装成唯一答案
- 若反思节点未通过，应回写修正后再生成终版

## 7. 语气规范

- 克制
- 具体
- 可执行
- 不神化历史
- 不道德审判用户
- 不夸大自己确定性

## 8. 前端返回建议

前端最终展示建议拆成结构化字段：

- `summary_block`
- `analysis_block`
- `evidence_cards`
- `strategy_block`
- `risk_block`
- `action_24h`
- `action_7d`
- `mermaid_graph`
- `poetry_comfort`

## 9. 验收标准

- 用户能快速读懂局势、策略、风险和下一步
- 每条关键策略都能追溯到至少一个历史映射依据
- 高情绪场景下先稳住语气，再给建议
- 整体输出像“决策幕僚报告”，而不是百科说明文
