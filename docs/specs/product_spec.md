# Product Spec

## 1. 产品定义

- 产品名：`资智 ZiZhi`
- 产品形态：本地 `Streamlit` Demo，后续可扩展为服务化应用
- 产品定位：基于《资治通鉴》的历史案例库与决策映射系统
- 核心价值：把现实组织/职场困局映射到历史案例结构，再给出可执行策略、风险提醒与克制表达

产品层与基础设施层需明确区分：

- 产品层：用户最终消费的是 `CaseProfile`
- 基础设施层：`chunk / retrieval` 负责供给证据与上下文

## 2. 目标用户

- 普通职场人：领导关系、同事竞争、边缘化、汇报沟通
- 中层管理者：团队冲突、跨部门协作、授权失衡、用人制衡
- 创业者/老板：合伙关系、控制权、组织变革、关键成员博弈
- 泛职场用户：去留进退、站队表态、权力边界、情绪压力

## 3. V1 核心能力

- 自由文本输入，不要求用户先做结构化填表
- 系统自动完成问题理解、意图识别、角色抽取、冲突识别
- 基于《资治通鉴》定位相关历史案例，并回查史料证据
- 输出结构化幕僚报告，而不是只返回一段自由回答
- 在高压情绪场景下，追加克制的诗词安抚

离线侧同步建设：

- `chunk -> case profile` 的案例生产流水线
- 后续补上 `cross-chunk case` 合并层

## 4. 标准输出

V1 输出固定包含以下部分：

- 局势概述
- 关键矛盾与角色关系
- 相关历史案例/史例证据
- 策略建议与风险提醒
- `Mermaid` 关系/博弈图
- 必要时的情绪安抚内容

## 5. 当前实现基线

- 前端：`app.py`
- 工作流编排：`zizhi/workflow.py`
- 节点逻辑：`zizhi/agents.py`
- 状态与输出结构：`zizhi/schemas.py`
- 检索：`zizhi/retrieval.py`
- 案例抽取：`scripts/batch_extract_case_profiles.py`

当前版本已具备：

- 会话级状态对象 `AnalysisState`
- LangGraph 可用时的图编排，失败时的手动串行兜底
- 关键词检索兜底，LanceDB + embedding 为可选增强路径
- 初步 `CaseProfile` schema 与离线构建链路

## 6. 非目标

V1 暂不做：

- 长期用户记忆
- 在线多租户服务
- 自动训练行业专属模型
- 完整史学考据系统
- 严格学术级引文排版

## 7. 成功标准

产品层面的 V1 验收标准：

- 用户输入能稳定转成结构化问题状态
- 至少能定位若干条“可解释”的历史案例或证据
- 策略建议能落到 24 小时 / 7 天行动层
- 输出语气克制，不夸大历史映射关系
- 对不确定信息保留边界，不装作确定

案例库方向的附加标准：

- 抽取出的 case 明显比裸 chunk 更适合现代迁移
- 不把纯编年事实硬包装成案例
- 对同一故事的重复 case 有后续去重与合并机制

## 8. 正式工程化方向

后续正式版本以以下原则推进：

- 产品主单元改为 `case`
- 主生产链按 `chunk -> case` 推进
- 检索语义主通道以白话文为主
- 古文原文与白话解释通过统一 `section_key` 关联
- chunk 切分从“解析段落”升级为“检索友好的 section-aware chunk”
- retrieval 优先服务案例命中，再做原文回查
- 语料、检索、输出格式均以 spec 为准，逐步替换 Demo 级实现
