# North Star Spec

## 1. 核心判断

`资智 ZiZhi` 的最终产品形态应当是一个基于《资治通鉴》的历史案例库，而不是一个以文本召回为中心的检索工具。

更准确地说：

- 产品层：`case library`
- 基础设施层：`section / chunk / retrieval`

## 2. 为什么不是“检索系统优先”

单纯检索更擅长回答：

- 哪段文字和我的问题相似
- 哪条史实与某个关键词相关
- 原文在哪里

但项目真正想回答的是：

- 有没有一个可迁移的历史案例
- 这个案例中的主导者、承压者、冲突与结果是什么
- 今天的人能从中吸收什么管理经验

所以：

- `retrieval` 解决“找得到”
- `case` 解决“用得上”

## 3. 项目分层

### 3.1 SourceSection

最小可追溯史料单元，负责原文与白话对齐。

### 3.2 RetrievalChunk

长度受控的检索窗口，负责召回、局部上下文和证据回查。

### 3.3 CaseProfile

真正面向用户消费的案例单元，负责表达：

- 谁在何种局面下做决策
- 冲突是什么
- 结果如何
- 今天能借鉴什么

### 3.4 Cross-Chunk Case

对长故事进行合并后的完整案例，用于处理单 chunk 难以承载的连续事件链。

## 4. 北极星标准

以后评估项目，优先问这几个问题：

- 这个 case 是否真的存在清晰的决策冲突
- 这个 case 是否比直接看 chunk 更有启发
- 这个 case 是否能迁移到现代管理/组织场景
- 是否把同一故事重复抽成多条
- 是否把一个应该完整的故事切得过碎

不要再把“能不能召回相似文字”当成最高指标。

## 5. 对工程方向的直接要求

- chunk 不再被视为最终产品单元
- case 成为主要入库与展示对象
- 正式主生产链按 `chunk -> case` 设计
- retrieval 改为优先服务 case 命中，再回查 chunk 证据
- answer generation 优先引用 case，再补充 section/chunk 证据
- 后续所有新 spec 都应服从这一分层

## 6. 非目标

本项目不追求：

- 把《资治通鉴》全文做成一个更强的古文搜索器
- 仅凭 embedding 相似度直接给管理建议
- 把每个 chunk 都强行包装成案例

## 7. 当前阶段结论

现阶段最值得投入的主线应是：

1. 提高 case 抽取质量
2. 做 case 去重
3. 做跨 chunk 合并
4. 让检索链路优先命中 case 而不是裸 chunk

如果这四件事持续推进，项目会越来越像“历史案例库”。
